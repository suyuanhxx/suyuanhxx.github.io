    缓存Redis

  5. redis中key的前缀生成策略如下：

    1. 基础方法：

    以项目+模块的方式进行分开。项目采用项目名称缩写，模块采用模块的缩写。
    举例：sdk行为的 SDK为项目缩写，ACTION为模块缩写，中间采用下划线 “_” 分割。则前缀key为 “SDK _ACTION” 。

    2. 高级方法

    对项目进行编码，采用16进展数。比如使用三位，对模块进行编码，比如使用三位。举例：sdk行为的，SDK编码：000，行为模块编码000，则前缀key为000000
    以基础方法命名key
    目前项目包括：

000000该编码，短小，唯一。能减少内存。只要维护好字典表，就能够快速的查询到对应的项目。为最优的解决办法。（仅供参考）

    系统架构

  3. 用户体系
  4. 常量结构
    背景，现在CommonConstants类是越来越庞大，而且现在微盘和大盘的分割会是两边后台代码维护越来越复杂。同时为了使代码更加简洁，清晰，优雅，也是为了提高代码可读性，降低耦合程度，提出以下几点归纳和建议：
    CommonConstants类现在主要包括：1.接口等加密所用的混淆“盐”值，2.账号类型常量值，3.部分操作具体小的业务常量值，4.redis所用到的key值。其中占大量篇幅的应该是redis的key值，重要的是账号类型的常量值。
    分解思路
    1.将redis key值抽出重新归纳为一个类中
    2.将所有账号类型相关的常量值统一使用枚举来实现，区分账号类型和业务类型。该类尽量减少其他代码的入侵。
    3.接口加盐加密抽离存放
    4.将具体的小业务常量存放在子项目中（比如文章相关的常量可以存放在tools中的common类中等）
